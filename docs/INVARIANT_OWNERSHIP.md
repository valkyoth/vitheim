# Invariant Ownership Registry

Status: normative planning registry introduced by `0.18.3`

This is the canonical index for authority-bearing invariants. Detailed phase
documents define behavior and test cases; they refer to stable IDs here instead
of creating a second owner. Before a milestone may exit, every invariant it
introduces or changes must have exactly one row, one authoritative owner, an
owner-maintained guard/update path, explicit transaction placement, semantic
storage requirements, all four test classes, and restore/migration obligations.

Every invariant has exactly one declaration marker in its introducing
milestone, in this form:
`<!-- vitheim-invariant VIT-INV-NNN 0.X.Y -->`. A milestone may introduce
multiple invariants and therefore contain multiple markers. The checker
discovers those markers rather than trusting a hand-maintained expected count,
then requires a matching ownership and lifecycle row in both directions.

The tables are deliberately machine-readable. Cells cannot be empty, contain
`TBD`, or name alternative owners. Each invariant has stable `VIT-ENF-*`
enforcement, `VIT-CAP-*` storage-capability, `VIT-TST-*` verification-contract,
`VIT-RCV-*` recovery-manifest, and `VIT-FEN-*` owner-transfer-fence IDs.
The required test markers are `P:` positive, `N:` negative, `M:` model/
interleaving, and `F:` fault/recovery. Storage requirements begin with
`requires:`; guard cells identify how the owner updates the guard with
`owner-updated:`; recovery cells contain both `restore:` and `migration:`.
`scripts/check_invariant_ownership.sh` enforces declaration coverage, identifier
resolution, row shape, and lifecycle symmetry; later implementation milestones
bind the stable contracts to concrete modules, conformance cases, storage
profiles, and generated recovery manifests.

## Governance And Change Rules

- Registry granularity is one row per authoritative ownership root. Multiple
  guards or fields updated by that same owner in the same transaction remain
  enforcement/storage contracts on its row; an independently updated state,
  separate owner, or separately restorable monotonic fact requires another row.
- A phase document may explain an invariant but cannot rename its owner or
  transaction domain without changing its declaration, ownership row, and
  lifecycle row in the same commit.
- A new enforcement point must add a negative test. A new storage profile must
  prove every referenced `VIT-CAP-*` capability used by its supported invariants
  or report the invariant unsupported during capability negotiation. Once code
  exists, each `VIT-ENF-*`, `VIT-TST-*`, and `VIT-RCV-*` ID resolves to concrete
  enforcement locations, conformance cases, and generated manifest fields.
- An invariant split or owner transfer is a migration: it needs a new stable ID
  and symmetric machine-readable supersession rows, an old/new owner fence,
  mixed-version behavior, migration contract, rollback floor, and exact-commit
  security review. The old row becomes `superseded`; history is never deleted.
- Backup, import, and migration tooling consumes this registry to build its
  monotonic-state manifest. Omitting a registered epoch, tombstone, guard,
  cursor, quota state, or recovery fact makes the operation fail closed.
- Phase O and `1.0.0` acceptance must report every selected production storage
  profile against every applicable invariant ID with no unowned or waived row.
