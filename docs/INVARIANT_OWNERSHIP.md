# Invariant Ownership Registry

Status: normative planning registry introduced by `0.18.3`

This is the canonical index for authority-bearing invariants and the latest
resolved view of composite laws. Historical law meaning is canonical in
`docs/LAW_GENERATIONS.md`; a latest row here never retroactively changes an
earlier milestone. Each generation is content-bound by the canonical
`docs/LAW_GENERATION_MANIFEST.md` schema and digest, admitted through
the `docs/LAW_MANIFEST_ADMISSIONS.md` planning superset and a
`docs/LAW_ACTIVE_CATALOGS.md` activation-floor artifact whose global lineage is
owned by `VIT-INV-057` and whose local admission/time/topology-observation
ratchets are owned by `VIT-INV-058`; `VIT-INV-059` independently owns its
rollout process manager, `VIT-INV-060` owns dynamic placement-topology
generations, and `VIT-INV-061` independently owns topology-mutation
authorization issuance from `0.141.0`.
The laws are bound to compiled transitions, outcomes, recovery, and tests by
`docs/LAW_SEMANTIC_REALIZATIONS.md`. Detailed phase
documents define behavior and test cases; they refer to stable IDs here instead
of creating a second owner. Before a milestone may exit, every invariant it
introduces or changes must have exactly one row, one authoritative owner, an
owner-maintained guard/update path, explicit transaction placement, semantic
storage requirements, all four test classes, and restore/migration obligations.

Every invariant has exactly one declaration marker in its introducing
milestone, in this form:
`<!-- vitheim-invariant VIT-INV-NNN 0.X.Y -->`. A milestone may introduce
multiple invariants and therefore contain multiple markers. The checker
discovers those markers rather than trusting a hand-maintained expected count,
then requires a matching ownership and lifecycle row in both directions.

The tables are deliberately machine-readable. A `VIT-INV-*` identifies one
authoritative ownership root, not an end-to-end rule assembled from several
owners. A cross-owner rule is instead a `VIT-LAW-*`: it names one coordinator,
all contributing invariant roots, every local linearization point, its
fail-closed state, and an end-to-end recovery proof. A coordinator records and
reconciles the law; it does not become an alternate owner of contributing
state.

Cells cannot be empty, contain
`TBD`, or name alternative owners. Each invariant has stable `VIT-ENF-*`
enforcement, `VIT-CAP-*` storage-capability, `VIT-TST-*` verification-contract,
`VIT-RCV-*` recovery-manifest, and `VIT-FEN-*` owner-transfer-fence IDs.
The required test markers are `P:` positive, `N:` negative, `M:` model/
interleaving, and `F:` fault/recovery. Storage requirements begin with
`requires:`; guard cells identify how the owner updates the guard with
`owner-updated:`; recovery cells contain both `restore:` and `migration:`.
`scripts/check_invariant_ownership.sh` enforces declaration coverage, identifier
resolution, row shape, composite-law declarations/lifecycle/dependencies,
explicit enforcement-to-negative mappings, lifecycle symmetry, version
ordering, and acyclicity across
every `docs/implementation/*.md` file, including production and future phase
documents. `scripts/check_law_generations.sh` separately proves that every
historical law generation is contiguous, uses only roots already effective at
that milestone, binds its exact coordinator/semantics/fences/migration/
rollback/contracts/manifest digest, accepts only meaningful successor deltas,
and resolves to the latest normalized view below.
`docs/AUTHORITY_REVIEWS.md` and its checker ensure every later milestone
explicitly addresses whether it creates or extends authority, blocks unresolved
proposals after planned status, and pins law extensions to the generation
effective at the milestone. Later
implementation milestones
bind the stable contracts to concrete modules, conformance cases, storage
profiles, and generated recovery manifests.

When `OrchestratorAttestedFencedLease` is selected,
`WorkloadLeaseActionAuthorityPortV1` is an explicit external authority-
capability boundary rather than another Vitheim invariant owner. The external
issuer exclusively owns claim IDs, lease generations, and monotonic sequences.
The existing invariant owner for each protected transition exclusively owns
`ConsumedWorkloadLeaseActionClaim` and commits it atomically with the protected
state/outcome. Stable request replay, action-digest mismatch rejection,
`WorkloadLeaseActionClaimIssuanceUnknown`,
`WorkloadLeaseActionClaimOutcomeUnknown`, no-reissue-until-reconciled,
expiry/revocation/failover, and externally evidenced restore high-watermarks
are mandatory wherever a row below admits that workload profile.

### Topology Authorization Replay Lifecycle Refinement

This refinement is normative for the VIT-INV-060 and VIT-INV-061 rows above
and introduces no shared authority owner. VIT-INV-061 owns its issuance
sequence, request/mutation uniqueness, budgets, hot issuance results and
issuer-side checkpoint. VIT-INV-060 owns consumption/expiry/deadline results
and its consumer-side checkpoint. Each owner compacts only its own local state;
authenticated archive exchange is evidence, never a cross-owner transaction or
authorization source.

`VIT-ENF-061-F` applies the frozen attempt-rate and outstanding-authorization
budget before allocating a monotonic `AuthorizationIssuanceSequence`.
`VIT-ENF-060-G` and `VIT-ENF-061-G` implement
`TopologyAuthorizationReplayLifecycleV1`: exact hot outcome through the frozen
horizon; authenticated predecessor-linked
`TopologyAuthorizationReplayCheckpointV1`; atomic covered-through
high-watermark advance before deletion; versioned set/archive commitments and
key rotation; bounded compaction/backpressure; and typed
`TopologyAuthorizationHistoricalStateUnavailable` when historical proof is
unavailable. A sequence at or below either owner's compacted high-watermark is
historical and can never become absent, fresh, reissuable or consumable.

`VIT-TST-060-N-G` and `VIT-TST-061-N-F/G` reject quota bypass, sequence reuse,
unbounded hot/backlog state, checkpoint fork/rollback/substitution, delete-
before-checkpoint, shortened exact horizon, unauthenticated key rotation,
archive loss interpreted as absence, and sensitive plaintext in the long-lived
checkpoint. Their M/F schedules race issuance and old replay against
checkpoint installation, hot deletion, coalescing, key rotation, crash,
failover, restore, migration and import. VIT-RCV-060/061 merge the greatest
sequence, checkpoint digest, covered-through high-watermark, key epoch and
compaction cursor, preserve every uncovered hot result, and fail closed for an
unprovable historical range.

## Explicit Enforcement-To-Negative Verification Map

Every semicolon-delimited ownership-matrix entry explicitly binds an immutable
`VIT-ENF-NNN-X` to its semantic enforcement-point description, matching
`VIT-TST-NNN-N-X`, and `active` or `retired` status. Textual order has no
meaning: reordering rows or entries cannot change a child ID's semantics. New
points receive unused suffixes; removed points remain as `retired` historical
entries with their old description and negative contract. Changing a
description behind an existing ID is a security-contract migration requiring
review and a new child ID. The checker rejects duplicate/missing/mismatched
child IDs and statuses; later realization indexes resolve every active child
pair to one concrete enforcement site and one direct fail-closed test.
Composite-law enforcement and recovery paths use their explicit
`VIT-LENF-*`/`VIT-LTST-*-N` contracts.

## Governance And Change Rules

- Registry granularity is one row per authoritative ownership root. Multiple
  guards or fields updated by that same owner in the same transaction remain
  enforcement/storage contracts on its row; an independently updated state,
  separate owner, or separately restorable monotonic fact requires another row.
- Cross-owner safety claims belong in the composite-law registry. Every law
  must resolve at least two ownership roots, exactly one coordinating invariant,
  local linearization points, a typed fail-closed state, and recovery that
  verifies every contributor without projection or search-index inference.
  Every law also has one declaration and lifecycle row. Changing contributors,
  linearization, failure semantics, or recovery advances the dependency/
  recovery contract and follows fenced, acyclic, version-ordered migration.
- A phase document may explain an invariant but cannot rename its owner or
  transaction domain without changing its declaration, ownership row, and
  lifecycle row in the same commit.
- A new enforcement point must add an explicit stable child ID, semantic
  description, negative child test, and status. A new
  storage profile must
  prove every referenced `VIT-CAP-*` capability used by its supported invariants
  or report the invariant unsupported during capability negotiation. Once code
  exists, each `VIT-ENF-*`, `VIT-TST-*`, and `VIT-RCV-*` ID resolves to concrete
  enforcement locations, conformance cases, and generated manifest fields.
- An invariant split or owner transfer is a migration: it needs a new stable ID
  and symmetric machine-readable supersession rows, an old/new owner fence,
  mixed-version behavior, migration contract, rollback floor, and exact-commit
  security review. The old row becomes `superseded`; history is never deleted.
  Successor versions strictly increase, successor rollback floors cannot precede
  predecessor activation, supersession graphs are acyclic, and `retired` means
  intentionally ended without a successor—not an alias for `superseded`.
- Backup, import, and migration tooling consumes this registry to build its
  monotonic-state manifest. Omitting a registered epoch, tombstone, guard,
  cursor, quota state, or recovery fact makes the operation fail closed.
- Phase O and `1.0.0` acceptance must report every selected production storage
  profile against every applicable invariant and composite-law ID with no
  unowned, undeclared, unresolved, or waived row.
